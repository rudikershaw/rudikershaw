Which Search Algorithm? - Part 2


In the first installation of Which Search Algorithm, I covered exhaustive and binary searches. I also briefly went through the required knowledge for getting the best out of these articles. This time I will be covering depth first and breadth first searches, so let's jump straight into the subject.


Depth and breadth first searches are searches performed on graph structures. For the purposes of this article I am going to assume we are dealing with an undirected graph in it's most abstract sense. We wont assume that we are using a graph of any particular type. As an example, if we assumed we were only dealing with trees we could make all kind of assumptions about not having to deal with cyclic paths. By assuming that anything goes in our graph we can theoretically write searches that will work on any kind of graph.


Before we talk about both searches we had better define our nodes so that we know what we are dealing with. I will leave out getters and setters for brevity (please remember encapsulation in your own code), as well as any validation we might want to do when constructing the nodes. Also the edges connecting the nodes will be implied by the connected nodes rather than being specifically defined.

Depth First Search


Performing a depth first search simply means following possible edges (or connections) in the graph to the end of the path first before going back and searching alternative routes. It is the computer science equivalent of looking for your keys by thoroughly searching each cupboard, draw, or shelf in a single room before moving onto the next room. Below are some simple examples of depth first searches.

Node depthFirstSearch(Node root, int value){
    Stack<Node> stack = new Stack<Node>();
    Set<Node> visited = new HashSet<Node>();
    if(root.value == value) return root;
    stack.add(root);
    while(!stack.empty()){
        Node node = stack.pop();
        visited.add(node);
        for(Node child : node.connections){
            if(child.value == value){
                return child;
            } else if(visited.contains(child)){
                continue;
            } else {
                stack.add(child);
            }
        }
    }
    return null;
}
function depthFirstSearch(root, value){
    var stack = [];
    var visited = [];
    if(root.value === value) return root;
    stack.push(root);
    while(stack.length > 0){
        var node = stack.pop();
        visited.push(node);
        for(var i = 0; i < node.connections.length; i++){
            if(node.connections[i].value === value){
                return node.connections[i];
            } else if(visited.indexOf(node.connections[i]) >= 0){
                continue;
            } else {
                stack.push(node.connections[i]);
            }
        }
    }
    return null;
}
def depthFirstSearch(root, value):
    stack = []
    visited = set()
    if root.value == value:
        return root
    stack.append(root)

    while stack:
        node = stack.pop()
        visited.add(node)
        for child in node.connections:
            if child.value == value:
                return child
            elif child in visited:
                continue
            else:
                stack.append(child)

    return null

Essentially the way this works is, starting with the root node in the graph, we keep track of each node we've looked at already and then we add the nodes who's connections we'll need to check later into a stack. The effect of this is that because stacks are last-in-first-out (LIFO) we end up dropping down a layer in the graph each time we check the next item in the stack. Breadth First Search


Determining whether you want to use a depth first or breadth first search depends on the nature of the data you are storing in your graph as well as the kind of information you want to get from the graph. Imagine a family tree expressed in graph form. If you wanted to perform a search for a living but distant relative in the graph you will be wanting to perform a breadth first search. If, on the other hand, you wanted to find an historic but direct ancestor from hundreds of years ago a depth first search makes much more sense.


This is what I mean by understanding something about the data and the nature of what you are looking for. If you have completely unstructured data with no pattern to the depth or breadth it doesn't make a great deal of difference which search type you choose. Conversely, knowing a little about the structure of your graph can tell you how you should be searching it. There is no hard and fast rules for choosing, it is about knowing your data, and knowing what you are looking for.
